# V0.3 Roadmap

Priority-ordered plan for the next release. Focus: making the proxy actually
proxy — body streaming, retries, keepalive, real TLS, signal handling, health
checks, and HTTP cache integration. v0.2 shipped all 13 package surfaces;
v0.3 makes them work together end-to-end.

---

## Phase 1 — Proxy production-readiness

### 1. Upstream response body streaming

Pingora crate: `pingora-proxy`. Currently the proxy sends the raw request bytes
to the upstream but does not read or forward the response body back to the
client. HTTP responses with bodies (anything beyond a `Content-Length: 0` or
`204 No Content`) will hang.

- [ ] `dispatchToUpstream` reads the upstream response status line + headers
- [ ] Stream response body chunks back to the downstream client as they arrive
- [ ] Handle chunked transfer-encoding (de-chunk or pass through)
- [ ] Handle `Content-Length` bounded body
- [ ] Handle `Connection: close` (read until EOF)
- [ ] Tests: echo upstream, verify body round-trip

### 2. Retry loop

Pingora crate: `pingora-proxy`. When an upstream connection fails, the proxy
should try the next peer (from LB) up to `max_retries`.

- [ ] `process_new` wraps the dispatch in a retry loop
- [ ] `upstream_peer` re-called on each retry to get a fresh peer
- [ ] `max_retries` configurable on `HttpProxy`
- [ ] `fail_to_connect` / `fail_to_proxy` / `error_while_proxy` callbacks fire
      on the final error (not on intermediate retries)
- [ ] Tests: point to closed ports, verify retry count and final error

### 3. Request body streaming

Pingora crate: `pingora-proxy`. POST/PUT requests with bodies must be forwarded
to the upstream.

- [ ] Read downstream request body (after headers) and forward to upstream
- [ ] `request_body_filter` callback for inspection/modification
- [ ] Handle `Transfer-Encoding: chunked` from client
- [ ] Handle `Content-Length` bounded body
- [ ] Tests: echo upstream with POST, verify body round-trip

## Phase 2 — Connection lifecycle

### 4. HTTP/1.1 keepalive

Pingora crate: `pingora-proxy`. Currently every request opens a new connection
to the upstream and closes it after one response. Keepalive reuses upstream
connections.

- [ ] `process_new` returns `?Stream` — when keepalive, return the stream for
      the accept loop to re-use on the same downstream connection
- [ ] Upstream connection pooling — reuse connections from `zigora_pool`
- [ ] `Connection: keep-alive` vs `Connection: close` negotiation
- [ ] Idle timeout on kept connections
- [ ] Tests: multiple requests on same downstream connection, verify single
      upstream connection used

### 5. Downstream keepalive + pipelining

Pingora crate: `pingora-proxy`. Serve multiple requests on a single downstream
TCP connection.

- [ ] `process_new` returns `Some(stream)` → `handleConn` passes it back
- [ ] Parse next request from the same stream (HTTP/1.1 pipelining)
- [ ] Limit: max N requests per connection (configurable)
- [ ] Tests: `nc` or `curl --parallel` to send multiple requests

### 6. ConnectionPool integration

Pingora crate: `zigora_pool`. The v0.1.1 `ConnectionPool` exists but is never
used by the proxy. Wire it in.

- [ ] `upstream_peer` tags connections with a `GroupKey`
- [ ] Before connecting, try `pool.get(key)` for a reusable connection
- [ ] After dispatch or on keepalive, `pool.put(key, connection)`
- [ ] Evict idle connections when pool is full
- [ ] Tests: verify connection reuse across requests

## Phase 3 — Server hardening

### 7. Signal handling + graceful shutdown

Pingora crate: `pingora-core`. Currently the server ignores SIGINT/SIGTERM and
must be killed with SIGKILL.

- [ ] `sigaction(SIGINT)` → set `shutdown_flag`
- [ ] `sigaction(SIGTERM)` → set `shutdown_flag`
- [ ] Accept loop polls `ShutdownWatch.check()` and exits when true
- [ ] In-flight requests get a grace period (`graceful_shutdown_timeout_ms`)
- [ ] After grace period, remaining connections are forcibly closed
- [ ] Tests: send SIGTERM, verify server exits cleanly with all services stopped

### 8. Structured logging

Pingora crate: `pingora-proxy` (logging callback). Currently the proxy logs
ad-hoc `std.log.info` messages.

- [ ] Structured log line per request: method, path, status, duration, bytes,
      upstream addr
- [ ] `logging` callback in `ProxyHttpVTable` fires for every completed request
- [ ] Error log lines include `ErrorSource` and retry count
- [ ] Configurable log format (JSON vs text) — stretch goal

## Phase 4 — Load balancing + health

### 9. Backend health checks

Pingora crate: `pingora-load-balancing`. Currently the LB selects from a fixed
set of backends regardless of whether they're alive.

- [ ] `HealthCheck` trait — `check(backend) → Result`
- [ ] `TcpHealthCheck` — dial with timeout, succeed on connect
- [ ] `HttpHealthCheck` — send `GET /health`, expect 200
- [ ] Background health check loop per backend (configurable interval)
- [ ] Unhealthy backends removed from active selection set
- [ ] Tests: start/stop upstream, verify LB skips dead backends

### 10. Backend discovery

Pingora crate: `pingora-load-balancing`. Currently backends are hardcoded at
startup.

- [ ] `Backends` type — thread-safe set of backends with `ArcSwap`-like update
- [ ] `add_backend(addr, weight)`, `remove_backend(addr)` at runtime
- [ ] API endpoint: `POST /api/backends`, `DELETE /api/backends/{addr}`
- [ ] DNS-based discovery — resolve hostname periodically, update backends
- [ ] Tests: add/remove backends at runtime, verify LB uses the new set

---

## Phase 5 — TLS for real

### 11. TLS accept (server-side)

Pingora crates: `pingora-openssl` / `pingora-boringssl` / `pingora-rustls`.

- [ ] Choose one TLS backend (rustls recommended — pure Zig-compatible bindings)
- [ ] `tls::accept(raw_stream, config) → TlsStream` — wrap accepted TCP stream
- [ ] `ServerConfig { cert_path, key_path, ca_path }`
- [ ] Listeners accept both TLS and plain TCP on different ports
- [ ] Tests: `curl --cacert cert.pem https://127.0.0.1:8443/`

### 12. TLS connect (upstream)

Pingora crate: `pingora-tls`.

- [ ] `tls::connect(io, addr, config) → TlsStream` — wrap outbound TCP stream
- [ ] Upstream peer can specify TLS vs plain per backend
- [ ] ALPN negotiation for HTTP/1.1 (and H2 in phase 6)
- [ ] Tests: proxy to TLS-enabled upstream

---

## Phase 6 — HTTP cache

### 13. HTTP cache integrated with proxy

Pingora crate: `pingora-cache`. The v0.2 `HttpCache` state machine exists but
is not wired into the proxy.

- [ ] `request_cache_filter` fires in `process_new` before upstream connection
- [ ] `response_cache_filter` fires after upstream response
- [ ] Cache storage backed by `zigora_memory_cache`
- [ ] Cache key = request method + host + path + vary headers
- [ ] `Cache-Control` header parsing (max-age, no-cache, no-store, must-revalidate)
- [ ] Stale-while-revalidate background refresh
- [ ] Tests: first request → miss + upstream fetch, second → hit, verify age header

---

## Phase 7 — HTTP/2

### 14. HTTP/2 server support

Pingora crate: `pingora-proxy` H2 server options.

- [ ] H2 frame parsing (settings, headers, data, goaway, ping)
- [ ] H2 connection preface + settings exchange
- [ ] Multiplexed request/response on a single TCP connection
- [ ] ALPN negotiation (requires TLS phase 5)
- [ ] Cargo-cult from Pingora's H2 implementation
- [ ] Tests: `curl --http2-prior-knowledge` or via TLS ALPN

Phases 5-7 add optional but important production capabilities.

## Acceptance criteria

```bash
# Proxy a real upstream with body
python3 -m http.server 9000 &
zig build run
curl -s http://127.0.0.1:8080/index.html | head -5  # returns HTML

# Retry on failure
# (start with no upstream, then start one)
curl -v http://127.0.0.1:8080/  # first: 502, then start upstream, retry: 200

# Keepalive
curl -v --http1.1 http://127.0.0.1:8080/  # Connection: keep-alive

# Graceful shutdown
kill -TERM $ZIGORA_PID  # server exits cleanly

# Health check
# start 2 backends, kill one, verify LB skips it

# TLS
curl --cacert cert.pem https://127.0.0.1:8443/

# Cached response
curl -s http://127.0.0.1:8080/  # first: miss
curl -s http://127.0.0.1:8080/  # second: hit (age header present)
```
